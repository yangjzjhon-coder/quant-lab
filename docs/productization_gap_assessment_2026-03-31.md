# Quant-Lab 产品化差距评估（2026-03-31）

这份文档回答的问题不是“代码还能怎么写”，而是“这个项目如果要真的落地成产品，还差什么”。

结论先写在前面：

- `quant-lab` 现在更像一个内部研究与 demo 执行系统
- 它已经有不错的共享 runtime 主干、研究治理和本地服务雏形
- 它离“内部可值守产品”已经不远
- 它离“对外公开售卖的 SaaS 产品”还很远

因此，默认产品化路径应当是：

1. 先内部产品化
2. 再单账户、单标的、单候选 Stage 0 live
3. 再考虑更大范围的生产化

不建议一开始就把目标设成对外 SaaS。

## 1. 已经具备的产品雏形

当前仓库已经具备几个很关键的产品骨架：

- 共享运行时与硬约束
  - 见 [docs/runtime_hard_constraints_2026-03-30.md](docs/runtime_hard_constraints_2026-03-30.md)
  - 共享决策源位于 `src/quant_lab/application/runtime_policy.py`
- 研究治理主干
  - 见 [docs/research_governance_workflow_2026-03-28.md](docs/research_governance_workflow_2026-03-28.md)
  - 已有 `task -> candidate -> evaluation -> approval -> bind/route`
- 本地服务和状态库
  - 已有 `FastAPI + SQLite/PostgreSQL`
  - 已有 runtime snapshot、heartbeat、alert event
- demo 执行与 preflight 思路
  - 已有 preflight、reconcile、submit gate、executor state
- Stage 0 live rollout 思路
  - 见 [docs/live_rollout_stage0_runbook_2026-03-30.md](docs/live_rollout_stage0_runbook_2026-03-30.md)

这说明它不是从零开始产品化，而是已经有一条主干可继续硬化。

## 2. 当前最现实的产品定位

最现实的产品定位不是“面向外部客户的量化平台”，而是：

- 面向内部使用者的研究与执行控制台
- 面向单账户的值守系统
- 面向受控范围的 Stage 0 live rollout 底座

如果接受这个定位，产品化重点应该是：

- 稳定
- 可值守
- 可审计
- 可恢复
- 可解释

而不是优先追求：

- 花哨 UI
- 策略数量
- 多租户
- 对外售卖能力

## 3. 离“内部可值守产品”还差什么

### 3.1 版本与发布管理

当前仓库能开发、能运行，但还需要更正式的发布机制：

- 统一版本号和变更记录
- 固定发布单位和回滚单位
- 版本对应的配置、文档、数据库解释同步
- 不同环境的配置模板和启动方式固化

如果没有这层，后面很难回答“现在线上跑的是哪一版”。

### 3.2 环境与密钥管理

当前已经有 `.env`、配置文件、profile 模板，但还需要更硬的环境治理：

- 开发、研究、demo、live 环境隔离
- 密钥和 profile 的加载规则固化
- 敏感配置不混入普通运行配置
- 启动前环境自检和错误提示标准化

### 3.3 可观测性与运维能力

产品化不等于页面能打开，而是出问题时能定位、能恢复。

还需要补：

- 更完整的健康检查与告警分级
- 统一的事件链路追踪
- 更明确的 operator 提示
- 日志归档与故障复盘材料
- 标准化 runbook

### 3.4 执行安全与恢复能力

这部分是从 demo 走向产品的核心差距。

还需要补：

- 更强的 reconcile 和交易所状态对账
- kill switch
- 执行异常后的自动降级或阻断
- protective stop 的恢复与重建逻辑
- 网络异常、重复提交、状态漂移的明确恢复流程

### 3.5 数据质量与结果可复现

产品化要求“同样输入能解释同样输出”。

还需要补：

- 数据版本标识
- artifact manifest 或统一产物索引
- 报表、回测、审批之间的引用关系
- 数据缺口与 fallback 的显式标注

### 3.6 数据库与状态演进机制

现在已经有状态表，但长期产品化还需要：

- migration discipline
- schema 变化的兼容策略
- 历史状态解释的稳定口径
- 环境间的备份和恢复流程

### 3.7 人机流程与角色分工

代码里已经有 `research_lead / strategy_builder / backtest_validator / risk_officer` 这些角色标签，但产品化还需要真正的操作流程：

- 谁创建 task
- 谁提交 candidate
- 谁评估
- 谁批准
- 谁负责启动/停止执行
- 谁处理告警和事故

如果这些角色只有表字段，没有操作流程，产品还是不完整。

### 3.8 UI 与操作体验

当前服务界面更像内部调试面板，不像成熟产品控制台。

内部产品化至少还需要：

- 更清晰的状态解释
- 更明确的失败原因展示
- 更直观的执行闸门显示
- 更统一的单标/组合视图
- 更明确的操作确认路径

## 4. 离“对外产品”还差什么

如果目标不是内部工具，而是对外产品，还要再加一整层能力：

- 账号体系
- 权限模型
- 多租户隔离
- 审计日志
- 计费与配额
- 客户支持流程
- 数据保留策略
- 法律与合规边界
- 风险揭示与责任划分

这条路和当前仓库的距离明显更远，默认不建议作为近期目标。

## 5. 建议的产品化推进顺序

### 阶段 A：内部产品化

目标：

- 让研究、审批、demo 执行、服务监控形成稳定闭环

重点：

- 基线稳定
- 文档完备
- 运行可解释
- 告警可处理
- 演练可复现

### 阶段 B：Stage 0 live rollout

目标：

- 单账户、单标的、单候选、单周期的最小 live 面

重点：

- 严格遵守 [docs/live_rollout_stage0_runbook_2026-03-30.md](docs/live_rollout_stage0_runbook_2026-03-30.md)
- live 配置冻结
- preflight / rollout policy / submit gate 全链路可解释
- 小规模真实运行可中止、可回退

### 阶段 C：内部生产化

目标：

- 从“能跑 live”推进到“能长期值守”

重点：

- 执行恢复
- 更强对账
- 更强观测
- 更强运维流程
- 更硬的版本与发布体系

### 阶段 D：更复杂的产品能力

目标：

- 多标的 live
- router-based live
- 多账户或多人协作

这一阶段不应早于前面三阶段。

## 6. 产品化决策默认值

如果没有额外商业要求，默认采用以下判断：

- 默认产品定位：内部产品，不是外部 SaaS
- 默认落地路径：先 demo 值守，再 Stage 0 live
- 默认 live 范围：单账户、单标的、单候选
- 默认优先级：稳定性和可值守性高于功能丰富度
- 默认 UI 目标：可解释、可操作、可审计，高于“好看”

## 7. 默认结论

`quant-lab` 已经具备“产品主干”，但还没有具备“成熟产品壳层”。

如果现在要认真推进产品化，最该做的不是再堆策略，而是补齐下面这条链：

`基线稳定 -> 运行约束 -> 研究治理 -> 执行门禁 -> 运维可观测 -> 发布与恢复 -> Stage 0 live`

这条链补齐以后，它会先成为一个靠谱的内部产品；那时再决定是否继续走向更复杂的外部产品。
